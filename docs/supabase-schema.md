# Supabase Schema — CathedralOS

This document describes the backend database foundation for CathedralOS:
generated story outputs, usage tracking, public sharing, and remix lineage.

---

## Why Postgres (not Supabase Storage) for text

Story text and structured metadata are stored directly in Postgres rows because:

- **Query-first**: filtering, sorting, and paginating outputs by user, date, or
  visibility requires relational queries that Postgres handles natively.
- **Atomic transactions**: writing the output row and a usage event in the same
  request is safe and straightforward with Postgres.
- **RLS is built-in**: row-level security policies attach directly to tables;
  applying equivalent logic to object-storage blobs would require extra
  indirection through Edge Functions or signed URLs for every read.
- **Size is small**: generated story text is typically 1–6 KB. Storing tens of
  thousands of outputs is well within Postgres's comfortable range.

Supabase Storage (or Cloudflare R2) is intentionally deferred to a later PR.
It will be used for user-uploaded assets, cover images, audio exports, and any
binary content that does not benefit from SQL queries.

## Tables

### `profiles`

Stores optional display metadata for each authenticated user.
Mirrors `auth.users` via a foreign key so the profile is automatically removed
when the auth record is deleted.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | References `auth.users(id)` |
| `display_name` | text | Optional, user-chosen name |
| `created_at` | timestamptz | Set on insert |
| `updated_at` | timestamptz | Auto-updated by trigger |

**RLS assumptions**
- Users can read, insert, and update only their own row.
- No public read; the profile is private by default.

---

### `generation_outputs`

The primary record of every story or content item generated by the AI.
Holds both the raw output text and a copy of the structured source payload
(the full `PromptPackExportPayload`) so the backend can reconstruct context
without depending on the iOS device's local SwiftData store.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | `gen_random_uuid()` |
| `user_id` | uuid FK | `auth.users(id)` |
| `local_generation_id` | text | Client-side UUID for dedup |
| `project_name` | text | Human-readable project label |
| `prompt_pack_name` | text | Prompt pack used |
| `title` | text | AI-generated or user-set title |
| `output_text` | text | Full generated story text |
| `source_payload_json` | jsonb | Complete `PromptPackExportPayload` |
| `model_name` | text | e.g. `gpt-4o` |
| `generation_action` | text | `generate` / `regenerate` / `continue` / `remix` |
| `generation_length_mode` | text | `short` / `medium` / `long` / `chapter` |
| `output_budget` | integer | Approximate token budget used |
| `status` | text | `draft` / `generating` / `complete` / `failed` |
| `visibility` | text | `private` / `shared` / `unlisted` |
| `allow_remix` | boolean | Whether others may remix this output |
| `created_at` | timestamptz | |
| `updated_at` | timestamptz | Auto-updated by trigger |

**Constraints**
- `generation_action` ∈ `{generate, regenerate, continue, remix}`
- `generation_length_mode` ∈ `{short, medium, long, chapter}`
- `status` ∈ `{draft, generating, complete, failed}`
- `visibility` ∈ `{private, shared, unlisted}`

**RLS assumptions**
- Full CRUD for the owning user only.
- No cross-user reads; sharing is handled via `shared_outputs`.

### `generation_usage_events`

Immutable audit log of every generation request, used for usage analytics and
future rate-limiting. One event is written per generation attempt regardless of
success or failure.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `user_id` | uuid FK | |
| `generation_output_id` | uuid FK nullable | Links to the resulting output if any |
| `action` | text | `generate` / `regenerate` / etc. |
| `model_name` | text | |
| `input_tokens` | integer | Reported by model |
| `output_tokens` | integer | Reported by model |
| `generation_length_mode` | text | |
| `output_budget` | integer | Requested budget |
| `status` | text | |
| `created_at` | timestamptz | |

**RLS assumptions**
- Users can read and insert their own events.
- No update or delete policies — usage records are immutable audit entries.
  Any corrections should be recorded as new rows.

### `shared_outputs`

A denormalized snapshot of a `generation_output` that is published for other
users to browse, favorite, or remix. Denormalization is intentional: it
decouples the public view from the author's private record and prevents
accidental data leaks when a private output is later modified.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `owner_user_id` | uuid FK | |
| `generation_output_id` | uuid FK nullable | Source output (set null on delete) |
| `share_title` | text | Author-chosen public title |
| `share_excerpt` | text | Short teaser shown in browse feed |
| `output_text` | text | Full text snapshot at publish time |
| `source_payload_json` | jsonb | Payload snapshot at publish time |
| `source_prompt_pack_name` | text | |
| `model_name` | text | |
| `generation_action` | text | |
| `generation_length_mode` | text | |
| `allow_remix` | boolean | Whether others may fork this output |
| `visibility` | text | `shared` / `unlisted` / `private` |
| `published_at` | timestamptz | When first published |
| `unpublished_at` | timestamptz | Set when owner unpublishes |
| `created_at` | timestamptz | |
| `updated_at` | timestamptz | Auto-updated by trigger |

**RLS assumptions**
- Authenticated users can read rows where `visibility in ('shared','unlisted')`
  and `unpublished_at is null`.
- The owner can read, insert, update, and delete their own rows regardless of
  visibility or unpublished state.
- Anonymous (unauthenticated) reads are not enabled by default; add a separate
  policy with `to anon` if a public browse-without-login feature is needed.

### `remix_events`

Tracks each time a user remixes a shared output. Used for lineage tracing,
analytics, and optionally notifying the original author.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `user_id` | uuid FK | User who performed the remix |
| `shared_output_id` | uuid FK nullable | Source shared output (set null on delete) |
| `created_project_local_id` | text | Client-side project ID of the new remix |
| `source_payload_json` | jsonb | Payload snapshot at remix time |
| `created_at` | timestamptz | |

**RLS assumptions**
- Users can read and insert their own remix events.
- No update or delete policies — remix events are immutable lineage records.

## updated_at trigger

A shared PostgreSQL function `public.set_updated_at()` is registered as a
`BEFORE UPDATE` trigger on tables that have an `updated_at` column:

- `profiles`
- `generation_outputs`
- `shared_outputs`

This ensures `updated_at` is always accurate without requiring the caller to
set it explicitly.

## Local setup

### Prerequisites

- [Supabase CLI](https://supabase.com/docs/guides/cli) ≥ 1.x
- Docker (required by `supabase start`)

### Start local Supabase

```sh
supabase start
```

This starts a local Postgres instance, Studio UI, and the auth stack. Your
local credentials are printed to the terminal.

### Apply migrations

```sh
# Against the local instance:
supabase db push

# Or step through migrations interactively:
supabase migration up
```

### Link to your hosted project (once)

```sh
supabase link --project-ref YOUR_PROJECT_REF
```

Then push to the hosted project:

```sh
supabase db push
```

### Reset local database (destructive)

```sh
supabase db reset
```

## Applying to a hosted project without the CLI

If you are using the Supabase Dashboard SQL editor directly, open the migration
file and run it in the **SQL Editor** of your project. All statements are
idempotent (`create table if not exists`, `create index if not exists`,
`create or replace function`).

## Migration validation notes

If the project does not have automated SQL tests yet, confirm the following
after applying the migration:

1. **All tables exist** — run `\dt public.*` in psql or check the Table Editor
   in Supabase Studio.
2. **RLS is enabled** — `select tablename, rowsecurity from pg_tables where
   schemaname = 'public';` should show `t` for every table.
3. **Policies exist** — `select tablename, policyname from pg_policies where
   schemaname = 'public' order by tablename;`
4. **Check constraints are enforced** — attempt an insert with an invalid enum
   value and confirm it is rejected.
5. **Triggers fire** — update a row on `generation_outputs` and confirm
   `updated_at` changes.

## Security notes

### Never embed the service-role key in the iOS app

The Supabase **service-role key** bypasses all RLS policies. It must only ever
be used in:

- Supabase Edge Functions (server-side, never shipped to the client)
- CI/CD scripts that need admin access
- Local development tooling

The iOS app receives only the **anon key** via `Info.plist`. All data access
is mediated by RLS policies. See `BACKEND_SETUP.md` for the iOS configuration
guide.

### RLS is your last line of defence

Even with RLS enabled, always validate inputs before writing them to the
database. The policies here enforce ownership; they do not validate the shape
or content of `source_payload_json`. Add application-level validation (e.g. in
Edge Functions) before writing user-supplied JSON to backend tables.
